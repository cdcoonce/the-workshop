# Build & Wiring Reference

How the plugin is assembled from source, and how hooks are wired into Claude
Code's lifecycle. The prose here is hand-written; the tables are generated by
`scripts/build_docs.py` from the actual scripts and settings, so they never drift
from what the repo does. Run `make docs` after changing a script or hook.

## The build model

The repo separates **source** from **output**:

- `core/` holds the universal skills, agents, hooks, methodology docs, and the
  base settings shared by every preset.
- `presets/<name>/` holds each preset's `manifest.json` (what to include,
  exclude, and add) plus any preset-specific skills, agents, hooks, and settings.
- `dist/<name>/` is the built plugin — self-contained and committed, so it can be
  installed directly by URL without a build step.

`build_preset.py` reads a preset's manifest, copies the selected core components,
layers the preset's own on top (override semantics: a preset component replaces a
core one of the same name), merges settings, and writes the plugin. `build_docs.py`
reads the same source metadata and writes this reference. Both are deterministic
and idempotent — re-running with no source change produces no diff, which is what
lets `make test` gate on staleness.

## Hook wiring

Hooks are Python scripts under `core/hooks/` (universal) or
`presets/<name>/hooks/` (preset-specific). A hook does nothing until it is
**wired** to a lifecycle event in settings: `core/settings-base.json` for the
universal hooks, and each preset's `settings-preset.json` for preset additions.
At build time the two are shallow-merged (hook arrays append, they don't
replace), and the merged wiring is written to the plugin's `hooks/hooks.json`.

The event a hook runs on is decided by its wiring, not its name — the table below
is derived from the settings, so it reflects reality even when a hook's docstring
uses a descriptive prefix (for example `protect-files.py` documents itself as a
"Pre-edit hook" but is wired to `PreToolUse`).

## Keeping this in sync

`build_docs.py --check` regenerates every doc in memory and fails if the
committed output differs. `make test` runs that check and rebuilds every preset
into `dist/`, so a change to any component that isn't accompanied by regenerated
docs and dist turns the build red. See the repo README's Development section for
the full loop.
