# Pack: UX for technical products

> Topic: Broad-SaaS + technical-product UX judgment — progressive disclosure, power-user patterns, tables/empty/error/loading states, onboarding for mixed audiences, docs-as-UX, AI-feature UX, good-enough-to-ship calls.
> Researched: 2026-07-19 · Staleness: 9 months default; entries marked **fast-moving** expire at 6 months (AI-product patterns churn).
> Load me when: arbitrating power-user vs newcomer tension (palettes, modes, settings sprawl); reviewing tables, empty/error/loading states, or error copy for a developer-adjacent audience; deciding onboarding shape for a mixed technical/non-technical user base, or whether docs/quickstart is the real onboarding surface; shaping AI features — review loops, confidence display, streaming, citations, ship gates; making ship/hold calls — severity ladders, debt ledgers, cut-vs-polish fights.

## Entries

### One lean surface, a palette, and config escape hatches — never a modes split

- **Claim:** Prefer one lean default UI plus a fuzzy-matching command palette over separate basic/advanced modes, and prefer burying advanced/rare/risky functionality behind CLI/config with documented escape hatches over growing the settings panel. A mode switch splits the product and forces users to self-identify; the palette absorbs the feature long tail; and if a setting needs a warning label, it belongs in config, not a checkbox. "More settings" ≠ "better usability."
- **When it applies:** Technical or mixed audiences, real feature depth to index, options that are advanced/rare/risky and documented. Only build the palette once there's a feature set worth fuzzy-matching.
- **When it fails / contested:** **Contested.** The "newcomers get discoverability" leg doesn't hold: palettes are recall-based, many newcomers never find Cmd+K, and Appleton's analysis says experts benefit far more than novices — the lean visible default carries the newcomer, not the palette. Don't ship hidden power-user behavior before the plain path is strong (uxpatterns.dev's own caution). The config-hatch move fails when you hide _common_ functionality or hide without docs — that's when the strip-the-options backlash critiques bite. Counter-source: Maggie Appleton, "Command K Bars" — https://maggieappleton.com/command-bar (2022); uxpatterns.dev command-palette cautions — https://uxpatterns.dev/patterns/advanced/command-palette
- **Source:** Evil Martians, "Six things developer tools must have in 2026" — https://evilmartians.com/chronicles/six-things-developer-tools-must-have-to-earn-trust-and-adoption (2026-01-06); Superhuman, "How to build a remarkable command palette" — https://blog.superhuman.com/how-to-build-a-remarkable-command-palette/ (2021-10-12)

### The three empty states are launch-blockers — and for technical users, the error state is diagnostic

- **Claim:** Prefer treating first-run, no-results, and error states as ship-blockers — each must say what happened plus what to do next — over spending that time polishing the populated view; the zero state does the persuasion work on day one while the populated view demos to nobody. And for developer-adjacent users, invert the consumer hide-the-detail heuristic: the error state carries a stable error code, request/trace ID, and documented fix path (Stripe's `req_` → dashboard log is the reference), not "something went wrong." Friendly-but-empty error copy is a support ticket; code + ID + link is an exit ramp.
- **When it applies:** Any container that can be legitimately empty; any surface where users must distinguish no-match from query-failed from not-loaded-yet; expected, documented error classes in a product with a technical segment.
- **When it fails / contested:** **Contested** on the diagnostic leg. OWASP's standing position is the opposite default — generic client-facing errors, details server-side only, no endorsement of returning correlation IDs — because verbose errors aid attacker recon. The resolution: diagnostic density for expected/documented error classes; deliberate vagueness for auth failures, enumeration-prone endpoints, and unexpected 500s, even in dev-facing products. On the empty-state leg: illustrations and delight copy are polish that can ship never; the blocker is what-happened + what-next, and the launch-blocker framing is a judgment call (directionally backed by the trust-erosion evidence, not itself studied). Counter-source: OWASP Error Handling Cheat Sheet — https://cheatsheetseries.owasp.org/cheatsheets/Error_Handling_Cheat_Sheet.html
- **Source:** NN/g (Kaplan), "Designing Empty States in Complex Applications" — https://www.nngroup.com/articles/empty-state-interface-design/ (2021-09-19); Pencil & Paper, "Empty State UX" — https://www.pencilandpaper.io/articles/empty-states; Stripe, "Error handling" — https://docs.stripe.com/error-handling; Rust RFC 1644 — https://rust-lang.github.io/rfcs/1644-default-and-expanded-rustc-errors.html (2016)

### Data-table MVP bar: sort, sticky header, frozen first column, basic filters — zebra only banned where rows have states

- **Claim:** Prefer shipping sorting + sticky header + frozen leftmost column + basic filtering as the table MVP, deferring density toggles, column management, and inline editing to fast-follow. The frozen identifying column is as load-bearing as the fixed header and is the piece teams routinely miss. In stateful tables, skip zebra striping — alternating fills collide with hover/selected/disabled into multi-grey noise; use light line divisions.
- **When it applies:** Interactive enterprise-style tables: many columns, horizontal scroll, row-level actions, scanning and comparison as the primary task.
- **When it fails / contested:** **Contested** on "skip zebra entirely." Enders' controlled study found striping caused no harm and was the plurality preference (46%), and she explicitly declined to rule against it; major design systems still ship zebra variants for wide read-only tables. Scope the ban to tables with row states; for dense read-only or very wide tables it's a legitimate row-tracking aid. Small static tables don't earn any of this machinery. Counter-source: Jessica Enders, "Zebra Striping: Does it Really Help?" — https://alistapart.com/article/zebrastripingdoesithelp/ (2008-05-06)
- **Source:** Pencil & Paper, "UX Pattern Analysis: Enterprise Data Tables" — https://www.pencilandpaper.io/articles/ux-pattern-analysis-enterprise-data-tables (2026-02-23) — attributed

### Responsiveness is a trust feature; stream generative output

- **Claim:** Prefer sub-100ms UI feedback (200ms upper comfort) and honest progress on slow jobs over shipping the next capability — technical users punish microfreezes harder than missing features, because latency reads as the product lying about its quality. For generative responses, prefer streaming partial output over any spinner or skeleton: perceived speed is time-to-first-token, and a blank pane that dumps a wall of text reads as broken. Skeletons are for structured payloads whose shape you know (tables, cards) — not token generation.
- **When it applies:** Daily-drive tools; roadmap fights where a 300ms input lag competes with a feature; any LLM-backed response a human reads as it arrives.
- **When it fails / contested:** Fake-optimistic spinners over honest progress is the anti-pattern in both halves — don't pretend slow jobs are fast, admit and show real progress. Don't stream machine-consumed or short structured JSON output; buffer-then-show anything a user might act on half-rendered. (Note: the "~3x faster perceived" number circulating for streaming isn't supported by its own source — Pockit's figure is ~40% faster perception; don't quote 3x.)
- **Source:** Evil Martians, "Six things developer tools must have in 2026" — https://evilmartians.com/chronicles/six-things-developer-tools-must-have-to-earn-trust-and-adoption (2026-01-06); Pockit, "Streaming LLM Responses in Web Applications" — https://pockit.tools/blog/streaming-llm-responses-web-guide/; The Prompt Bench, "Streaming UI Patterns That Don't Break" — https://thepromptbench.com/ai-product-ux/streaming-ui-patterns-that-dont-break/ (2026-05)

### Mixed-audience onboarding: value before prerequisites, instrument the verdict window, make every question buy a branch

- **Claim:** When activation has a technical gate (API key, integration, data import), prefer sandbox/sample-data/pre-populated value-first flows over prerequisite-first screens — the evaluator is often not the person who can complete the prerequisite, and a first-run blocked on IT is the anti-pattern. For the technical half, the docs and first runnable example ARE the onboarding surface: instrument time-to-first-working-result (the 5–10 minute keep/abandon window) — they leave silently, no rage-click telemetry. And only ask "what will you use this for" when the branches genuinely diverge: if you can't name what the developer branch shows that the ops branch doesn't, the question is suspect.
- **When it applies:** Products sold to one persona and configured by another; any product where a meaningful segment goes straight to the quickstart and never sees the tour.
- **When it fails / contested:** **Contested** on two legs. (1) Defunding in-app guidance overreaches for the non-technical half: vendor benchmark data (Chameleon — interested party, but the only data) shows checklists + tooltips doubling activation; for genuinely mixed audiences do both — TTFWR for the docs path, lightweight guidance for the rest. (2) A segmentation question can earn its friction without divergent in-app flows: answers feed CRM/lifecycle personalization and act as micro-commitments — so the delete-the-question test is a prior, not a law. Also: the "~60% onboarding time drop" number sometimes attached to sandbox claims is an unverified vendor anecdote — don't cite it. Counter-sources: Chameleon, "Effective Product Tour Metrics" — https://www.chameleon.io/blog/effective-product-tour-metrics; Appcues, "User Onboarding Best Practices" — https://www.appcues.com/blog/user-onboarding-best-practices; Userpilot, "SaaS Welcome Surveys" — https://userpilot.com/blog/welcome-survey/

### Fund docs as a product surface; use Diátaxis as a lens; structure survives the bot

- **Claim:** The moment any meaningful segment self-serves, prefer docs with an owner, a roadmap, and telemetry over docs as a content backlog — the transferable Stripe move is the accountability structure (someone owns docs outcomes the way a PM owns activation) plus writing culture wired into ladders, not the budget. Use Diátaxis as a diagnostic lens to catch mixed-mode pages, not as your nav — four literal top-level buckets fragment a single journey and punish non-technical readers most. And don't let "the bot will answer it" excuse bad IA: a chat widget on messy docs inherits the mess; structure is shifting from a human-nav concern to a machine-consumption concern, not going away.
- **When it applies:** Any meaningful self-serve segment; docs restructures; the temptation to bolt RAG chat onto an unmaintained docs pile.
- **When it fails / contested:** Both docs-investment sources are vendors with an interest in the story — the mechanism is corroborated, the ROI framing is theirs. On Diátaxis: Sequin (developer-only audience) did adopt roughly four Diátaxis buckets as nav and reports success — the lens-not-nav rule is scoped to mixed audiences, and even Diátaxis' own site says it isn't "four boxes." The AI-interface leg is the fast-moving part — re-check how agents consume docs at 6 months even though the IA principle is durable.
- **Source:** Mintlify, "How Stripe creates the best documentation" — https://www.mintlify.com/blog/stripe-docs (2023-11-10); Moesif, "Stripe Developer Experience Teardown" — https://www.moesif.com/blog/best-practices/api-product-management/the-stripe-developer-experience-and-docs-teardown/ (2026-05); Tom Johnson, "What is Diátaxis?" — https://idratherbewriting.com/blog/what-is-diataxis-documentation-framework (2023-10-18); Diátaxis, "Complex hierarchies" — https://diataxis.fr/complex-hierarchies/

### Ship llms.txt for agents, skip it for AI-search visibility

- **Claim:** Prefer treating AI agents as a first-class docs audience — llms.txt / llms-full.txt, clean markdown surfaces — over treating llms.txt as an SEO play. Google's May 2026 guidance is explicit that llms.txt does nothing for AI Overviews/AI Mode; meanwhile Mintlify's July 2026 benchmark (2,400 directed agent runs across Claude Code/Codex, replicated across frontier models) shows markdown + llms.txt cutting dead URLs ~90% and reducing token burn vs HTML. Audience resolves the debate: agents integrating against your product → ship it; chasing AI-search citations → cargo cult.
- **When it applies:** Agent-facing product docs; anything Claude Code/Cursor-class tools get pointed at during integration.
- **When it fails / contested:** **Contested.** The benefit currently depends on agents being _pointed_ at it — Mueller likens llms.txt to the keywords meta tag, and no AI system fetches it automatically; it's a community convention with no standards-body status. And the "nearly half of docs traffic is AI agents" stat circulating with this claim is unverifiable — the only sourced figure is Vercel attributing ~10% of new signups to ChatGPT referrals; don't repeat the half claim. Counter-sources: Passionfruit, "Should I Create an llms.txt File? Google's 2026 Guidance" — https://www.getpassionfruit.com/blog/should-i-create-an-llms.txt-file-google-s-2026-guidance-explained (2026-06-25)
- **Source:** Mintlify, "Docs URL Benchmark: Markdown & llms.txt > HTML" — https://www.mintlify.com/blog/llms-txt-agent-benchmark (2026-07-17); Passionfruit (above, confirms Anthropic/OpenAI/Perplexity adoption + Google's position)

### Bolt-in over bolt-on: task-shaped AI surfaces, not a chat sidebar

- **Claim:** Prefer embedding AI as task-shaped surfaces — inline suggestions, draft states, structured actions — over a chat sidebar, unless the product's core job is genuinely open-ended. The sidebar was cargo-culted from ChatGPT, whose whole product IS chat; focused workflows want focused UI. Chat wins breadth-first exploration and loses everywhere the user already knows what artifact they're changing.
- **When it applies:** Adding AI to a product with defined workflows and known artifacts (CRM records, documents, queries, configs).
- **When it fails / contested:** Genuinely open-ended, long-tail intent — exploratory Q&A, brainstorming — where chat legitimately wins; the carve-out is real, not a fig leaf. The pragmatic counter from the market: chat's zero learning curve drove adoption and the labs retrofitted GUI around chat rather than abandoning it. (The "~71% of AI-feature abandonment is interaction design" stat traces to an unverifiable source — don't cite it.)
- **Source:** Mutalik, "Beyond Chatbots: 5 Emerging AI-UX Patterns" — https://medium.com/@srivatsmutalik909/beyond-chatbots-5-emerging-ai-ux-patterns-every-designer-must-know-in-2025-9226f88f08e3 (2025-07); Jakob Nielsen, "UX Roundup: 2025 Predictions Revisited" — https://jakobnielsenphd.substack.com/p/ux-roundup-20251222 (2025-12-22)
- **Date:** Sources 2025 · researched 2026-07-19 · **fast-moving: 6 months**

### Trust UI must do something: categorical confidence mapped to behavior, verification at the point of claim

- **Claim:** Prefer categorical confidence ("Likely"/"Uncertain") mapped to a concrete behavior (auto-escalate, demand review) over raw percentages — users read "72%" as calibrated probability when the model means "guessed," and a single moderately-high score actively manufactures trust. And prefer point-of-claim verification affordances — inline citations, source counts, self-consistency flags — over blanket "AI can make mistakes" disclaimers, which are background clutter within a week. In both cases the test is the same: if the trust signal doesn't change what the UI or the user does, it's decoration.
- **When it applies:** Any AI output feeding a decision; any urge to decorate predictions with a percentage or to bolt a disclaimer under a chat box.
- **When it fails / contested:** **Contested** on both legs. (1) Zhang et al. found raw numeric confidence helped _untrained_ lay users calibrate trust — the failure mode is miscalibration and false precision, not numerals per se; and verbal labels carry their own inconsistent-interpretation problem. Genuinely calibrated, monitored systems in expert or regulated contexts can and sometimes must show numbers. (2) Citations are only a verification affordance if they're faithful: measured deployments had ~a quarter of citations failing to support their sentence, and users rarely click through — unfaithful citation UI manufactures trust a disclaimer never could. Ship attribution QA, not just citation chrome. Counter-sources: Zhang, Liao & Bellamy — https://arxiv.org/abs/2001.02114 (2020); Liu, Zhang & Liang, "Evaluating Verifiability in Generative Search Engines" — https://arxiv.org/abs/2304.09848 (2023-04)
- **Source:** NN/g, "AI Hallucinations: What Designers Need to Know" — https://www.nngroup.com/articles/ai-hallucinations/ (2025-02-07); TianPan, "The Accuracy Threshold Problem" — https://tianpan.co/blog/2026-04-10-accuracy-threshold-ai-features-70-85-percent-zone (2026-04-10); Lu et al., miscalibrated-confidence study — https://arxiv.org/abs/2402.07632 (2024)

### The agent change loop: editable plan before, diff-plus-reason after, edit-in-place, apply/revert — risk-tiered

- **Claim:** For AI that edits or acts with side effects, prefer the full loop over any single gate: an editable plan before execution (logs explain failures; plans prevent them), diff-plus-reason for edits (the reviewer never spots the change themselves), edit-then-approve draft states (fix the wrong 10% instead of reject-and-regenerate), and apply/revert inside the existing undo/review system — never execute model output directly, ship opt-in behind flags first. Version/hash the proposal so what was approved is what executes.
- **When it applies:** Agent workflows with side effects; AI edits to existing content; anything entering a system of record.
- **When it fails / contested:** **Contested**, three ways. (1) Scale: per-edit review becomes the bottleneck as agent throughput grows — "code got cheaper, judgment did not" — so the durable shape is risk-tiered approval (auto-apply low-risk with rollback/checkpoints, human confirmation reserved for irreversible/high-blast-radius), not preview-diff-everything; both sides agree irreversibles need explicit confirmation. (2) Diff-only review is a known blind spot — Google's reviewer guidance says sometimes you must read the whole file/system to know the change makes sense; the diff is the entry point of review, not its boundary. (3) "Approve must be the cheapest action" backfires: cognitive-forcing research shows friction on acceptance _reduces_ overreliance — cheap approval is the mechanism behind blind batch-approve. And binary approve/reject isn't legacy; it's still the shipped primitive in major agent infra (Cloudflare). Counter-sources: The Butler, "Coding Agents, Decision Fatigue, and the New Review Bottleneck" — https://thebutler.tech/2026-05-24-coding-agents-decision-fatigue-review-bottleneck/ (2026-05-24); Google eng-practices, "What to look for in a code review" — https://google.github.io/eng-practices/review/reviewer/looking-for.html; Buçinca, Malaya & Gajos — https://arxiv.org/abs/2102.09692 (2021); Cloudflare Agents HITL docs — https://developers.cloudflare.com/agents/concepts/agentic-patterns/human-in-the-loop/ (2026)

### Gate AI ship decisions on reversibility — then price the verification tax

- **Claim:** Prefer gating AI-feature ship calls on reversibility, not accuracy: ship the 70–85%-accurate feature on reversible, review-expected tasks (drafts, suggestions), hold anything irreversible to near-human accuracy. The 70–85% zone is the trap — good enough to look like it works, unreliable enough to burn trust once users stop checking; the failure isn't shipping there, it's shipping _as if you were at 95%_. Per-task accuracy breakdowns over one aggregate number; confidence-threshold auto-escalation over hoping users ask for help.
- **When it applies:** Any AI-feature go/no-go where someone is quoting a benchmark number as the ship criterion.
- **When it fails / contested:** **Contested.** Reversibility bounds downside, not net value: METR's RCT put experienced devs on a textbook reversible, review-expected task class (code drafts under review) and they finished 19% _slower_ with AI — while believing they were 20% faster. "Reversible + reviewed" is not license to ship 70–85%; the human verification tax must be measured, not assumed cheap, and self-reported benefit is unreliable. Counter-source: METR, early-2025 developer-productivity RCT — https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/ (2025-07-10)
- **Source:** TianPan, "The Accuracy Threshold Problem" — https://tianpan.co/blog/2026-04-10-accuracy-threshold-ai-features-70-85-percent-zone (2026-04-10) — attributed

### A named severity ladder gates the release; only red-route failures block

- **Claim:** Prefer an explicit, named severity ladder with only the top rungs blocking (Netlify's mountain/boulder/pebble/sand/dust; Nielsen's tiers) over ad-hoc "I feel strongly" crit weight — the label does the political work, letting a senior register the pebble without spending ship-date capital, and stopping the loudest reviewer from becoming the de facto scale. Gate on red-route task completion and recoverability: can users on frequent/critical paths finish, and climb out when they stumble. A release with only cosmetic-tier issues outstanding ships; the issues only designers notice are, definitionally, not on the blocking side.
- **When it applies:** Pre-ship reviews, design crits with release stakes, any recurring blocking-vs-nit re-litigation.
- **When it fails / contested:** Severity assignments from a single rater are unreliable — Nielsen himself says single-evaluator ratings can't be trusted and recommends aggregating 3+ — so the ladder needs cross-functional assignment, not solo verdicts. That critiques how rungs get assigned, not the ladder itself; ad-hoc feeling-strength is strictly less reliable. Three separate traditions (design crit, usability eval, code review) converge on the same two-tier shape, which is why this one is durable.
- **Source:** swyx, "Feedback Ladders" (Netlify) — https://dev.to/swyx/feedback-ladders-i (2020-03-08); Nielsen, "Severity Ratings for Usability Problems" — https://www.nngroup.com/articles/how-to-rate-the-severity-of-usability-problems/ (1994-11-01); Graphite, "Blocking comments vs. nitpicks" — https://graphite.com/guides/blocking-comments-vs-nitpicks

### File everything, fight selectively, fund the ledger

- **Claim:** Prefer "ship now plus a scored debt ticket that gets real sprint capacity" over blocking launch on polish or trusting an unbudgeted fast follow — a fast follow without a backlog slot is a euphemism for never. Record every deviation cheaply in a centralized debt ledger; pick which to _fight for_ by effort×impact with eng in the room — the designer who escalates everything gets nothing fixed, and severity is a cross-functional call (one person's polish issue is another's usability blocker). The ledger is what makes "ship with cosmetic issues" defensible instead of corrosive.
- **When it applies:** Every launch with known non-blocking issues; design-QA passes on built work; recurring debt triage.
- **When it fails / contested:** **Contested** on where selectivity lives: the debt literature (including the same authors) says log _every_ instance — selectivity at fix-time, not file-time; read as "don't record everything," the battle-picking advice contradicts its own evidence base, because unrecorded deviations become invisible compounding drift. Also: the specific 10–20% capacity figure and its "40% faster delivery" payoff trace to a single vendor post with no methodology — treat the number as practitioner heuristic; the multi-source consensus covers "scored ledger + real recurring capacity," not the percentage. Counter-sources: Chung/LogRocket — https://blog.logrocket.com/ux-design/design-debt-is-slowing-you-down (2025-05-28); Koru UX — https://www.koruux.com/blog/product-managers-overcome-ux-debt/
- **Source:** Chung/LogRocket (above); Figr, "How to Reduce Design Debt" — https://figr.design/blog/how-to-reduce-design-debt (2026-05-17); Shane Allen, "Design Quality (QA)" — https://shaneallen.medium.com/design-quality-qa-92d2a9a6ba34 (2021-01-02)

### Deadline triage: judge by the outcome, cut scope to hold the bar, kill the sunk feature, shrink the slice

- **Claim:** When the deadline squeezes, prefer cutting scope to protect the bar over thinning craft across everything — quality is the fixed minimum, so the honest negotiation is "do fewer things at the bar." If a feature has slipped so far it can't be made good in the time it's worth, kill it (Zhuo's test: "if we'd known it would take this long, would we have started?" — a no means cut, not degrade). Judge "good enough" against the business outcome the release must move, not fidelity to the spec — ship outcomes, not design. And when the full design is blocked by org reality, manufacture the smaller shippable step and champion it rather than filing the blocked mock as a grievance.
- **When it applies:** Post-discovery teams shipping to real users; ship-vs-polish fights the severity ladder alone doesn't settle; hard org or deadline blocks on the ideal design.
- **When it fails / contested:** **Contested**, three ways, and the boundaries matter. (1) Pre-product-market-fit, the lean school says the bar itself is the variable — ship rough to maximize validated learning; cut-vs-degrade is the right frame only once you know the feature is worth building. (2) The craft-as-strategy school (Saarinen/Linear: "quality creates gravity") argues that in quality-differentiated markets, fidelity IS the outcome lever — "ship outcomes, not design" becomes a standing excuse for erosion; know which market you're in. (3) The local-maximum critique: habitual slice-shrinking plateaus at the limit of the current design, and a shipped smaller slice gets treated as done — know when to stop slicing and hold out for the structural change.
